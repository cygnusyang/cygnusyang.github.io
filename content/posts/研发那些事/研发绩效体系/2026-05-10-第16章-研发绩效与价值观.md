前面几篇讨论了 KPI、研发绩效逻辑、方法论、企业类型和初创团队实践。到这里还需要补上最后一层：

**价值观。**

价值观看起来像一个很软的话题，但在研发绩效里，它其实是很硬的一层。因为研发工作面对大量不确定性，KPI 不可能提前定义所有正确行为，流程也不可能覆盖所有复杂判断。

当指标不够用、流程没有答案、结果还没有出现时，团队靠什么判断一件事该不该做、该怎么做、做到什么程度？

答案就是价值观。

## 为什么价值观重要

在研发绩效体系里，KPI 和价值观解决的是两个不同问题。

```text
KPI 解决：我们观察什么
价值观解决：什么行为值得鼓励，什么行为不能被奖励
```

如果只有 KPI，没有价值观，团队很容易围绕指标博弈。

考核交付率，团队可能降低需求难度；考核缺陷数，团队可能减少暴露问题；考核上线次数，团队可能拆碎发布；考核成本，团队可能牺牲长期质量。

这些问题不是 KPI 本身造成的，而是因为 KPI 缺少价值观约束。

价值观的作用，是给绩效体系提供行为边界。它告诉团队：即使某种行为短期能让指标变好，也不一定应该被奖励。

## 研发的不确定性，需要价值观补位

研发不是简单执行任务，而是把不确定性问题通过探索、验证和工程实现，转化为可靠能力。

这类工作有几个天然特点：

1. 很多重要行为无法提前量化；
2. 很多价值要过一段时间才体现；
3. 很多正确选择短期看起来不一定好看；
4. 很多错误行为短期可能会制造漂亮结果。

例如：

| 场景 | 只看结果可能怎么判断 | 加入价值观后怎么判断 |
|------|----------------------|----------------------|
| 团队验证后发现路线不可行 | 项目失败 | 高质量探索，避免更大浪费 |
| 团队按期上线但留下严重技术债 | 项目成功 | 短期完成，长期透支 |
| 团队主动暴露风险导致延期 | 进度不好 | 透明负责，降低系统风险 |
| 团队隐瞒缺陷按期交付 | 进度很好 | 低质量行为，不应奖励 |

所以，研发绩效不能只问“结果如何”，还要问“这个结果是用什么方式取得的”。

价值观就是用来回答后一个问题的。

## 价值观在绩效体系中的位置

前面我们把研发绩效体系拆成几层：

```text
目标 -> 阶段 -> 绩效结构 -> 证据 -> KPI
```

但如果要更完整，还应该在下面加一层：

```text
价值观边界
```

也就是：

```text
目标 -> 阶段 -> 绩效结构 -> 证据 -> KPI
                    ↓
                价值观边界
```

价值观不替代 KPI，也不替代结果评价。它的作用是约束绩效判断：

1. 哪些结果值得奖励；
2. 哪些行为不能因为结果好就被奖励；
3. 哪些失败不应该被惩罚；
4. 哪些短期低效其实是在保护长期能力。

## 研发团队最重要的价值观

不同公司会有不同的价值观表达，但从研发绩效角度看，真正关键的不是口号，而是下面几类行为准则。

### 1. 求真

研发首先要尊重事实。

求真意味着：不为了证明自己正确而选择性使用数据，不为了保住方案而隐瞒风险，不为了绩效好看而包装结论。

可观察行为：

| 高质量行为 | 低质量行为 |
|------------|------------|
| 主动暴露问题和不确定性 | 隐瞒风险，等问题最后爆发 |
| 用数据、实验、评审支持判断 | 靠拍脑袋和权威压制讨论 |
| 证据不足时承认不知道 | 把猜测包装成结论 |
| 发现路线错误后及时调整 | 明知有问题仍硬推原方案 |

研发绩效里最该保护的，是愿意说真话的人。

### 2. 客户价值或问题价值

研发不是为了完成内部任务，而是为了解决真实问题。

这里的“客户”可以是外部用户、业务方、内部平台使用者，也可以是某个真实工程问题的承担方。关键是：研发工作必须对准真实价值。

可观察行为：

| 高质量行为 | 低质量行为 |
|------------|------------|
| 追问问题是否真实、是否重要 | 只接任务，不问价值 |
| 用用户反馈和业务结果校准方向 | 用内部完成度替代外部效果 |
| 愿意砍掉低价值功能 | 为了显得忙而堆功能 |
| 优先解决关键瓶颈 | 长期做边角优化回避难题 |

如果没有客户价值或问题价值，研发绩效很容易变成内部自嗨。

### 3. 责任感

责任感不是加班，也不是服从。

研发里的责任感，是愿意对问题闭环负责，而不是只对自己手上的任务负责。

可观察行为：

| 高质量行为 | 低质量行为 |
|------------|------------|
| 主动推动问题闭环 | 任务交出去就不管后果 |
| 看到跨团队问题愿意协同解决 | 只强调边界，不承担共同结果 |
| 出现事故后复盘和改进 | 只解释责任不在自己 |
| 对长期质量负责 | 用短期交付掩盖长期成本 |

一个研发团队有没有责任感，往往体现在问题没人明确分配时，是否有人愿意站出来。

### 4. 长期主义

研发很容易被短期交付压力牵引。

长期主义不是不要短期结果，而是不允许用破坏长期能力的方式换短期好看。

可观察行为：

| 高质量行为 | 低质量行为 |
|------------|------------|
| 有意识管理技术债 | 技术债不可见、不可控、不可偿还 |
| 沉淀组件、文档、经验和工具 | 每个项目从头再来 |
| 在速度和质量之间做清晰权衡 | 用“赶进度”掩盖质量问题 |
| 为未来复用投入必要成本 | 只追求当期完成率 |

长期主义是研发绩效里最容易被忽视，但最影响组织后劲的价值观。

### 5. 协作

研发问题越来越少是单人问题，越来越多是系统问题。

协作不是开更多会，而是让信息、判断、风险和能力在组织内流动。

可观察行为：

| 高质量行为 | 低质量行为 |
|------------|------------|
| 主动同步关键风险和决策 | 信息只在小圈子里流动 |
| 帮助上下游理解约束 | 把复杂问题甩给别人 |
| 建设可复用能力 | 每个团队重复造轮子 |
| 有分歧时回到事实和目标 | 用职位、资历或情绪压制讨论 |

协作价值观的核心，是把团队从“各自完成任务”带向“共同解决问题”。

## 价值观如何进入绩效评价

价值观不能只停留在墙上，也不能只在年会上讲。

它必须转成可观察行为。

一个可用的方法是：

```text
价值观 -> 行为标准 -> 证据 -> 绩效判断
```

例如：

| 价值观 | 行为标准 | 证据 |
|--------|----------|------|
| 求真 | 主动暴露风险，基于证据调整方案 | 风险清单、验证报告、评审记录 |
| 客户价值 | 解决真实问题，而不是只完成任务 | 用户反馈、客户结果、业务指标 |
| 责任感 | 推动问题闭环，不甩锅 | 复盘记录、缺陷闭环、协同反馈 |
| 长期主义 | 管理技术债，沉淀复用能力 | 技术债台账、文档、组件、自动化 |
| 协作 | 促进跨团队问题解决 | 项目复盘、上下游反馈、共同交付结果 |

这样做的目的，不是把价值观重新变成一堆机械指标，而是让价值观评价有事实基础。

## 价值观和低绩效

价值观也能帮助我们重新理解低绩效。

低绩效不应该是一个配额，而应该是诊断结果。更准确地说，低绩效首先应该指向低质量行为。

例如：

1. 无验证、拍脑袋；
2. 隐瞒风险；
3. 只追求完成率，不关心真实效果；
4. 反复制造质量问题；
5. 不复盘、不闭环；
6. 为了短期结果透支长期能力；
7. 破坏协作，让团队系统性变慢。

这些行为即使短期结果不错，也不应该被评价为高绩效。

反过来，一个探索项目失败了，但团队有清晰假设、有效验证、真实数据、明确结论和及时止损，这不应该被简单打成低绩效。

这就是价值观对研发绩效最重要的修正：

**它把“结果好不好”推进到“行为质量好不好”。**

## 价值观考核的风险

价值观重要，但价值观考核也很容易被用坏。

常见问题有三类。

### 1. 口号化

价值观写得很漂亮，但没有行为标准，也没有真实案例。

这种价值观不会进入绩效，只会变成宣传语。

### 2. 道德审判化

价值观评价如果只靠主观印象，很容易变成给人贴标签。

例如“这个人不够积极”“这个人价值观不好”。如果没有具体行为和事实证据，这类评价会伤害信任。

### 3. 权力工具化

价值观如果只由管理者单向解释，就可能变成权力工具。

真正好的价值观评价，必须允许被讨论、被质疑、被事实校准。否则它会比 KPI 更危险。

所以，价值观进入绩效体系时必须遵守一个原则：

**评价行为，不评价人格；基于事实，不基于好恶。**

## 初创团队怎么做

初创团队不需要一开始就写很多价值观。

价值观越多，越容易空。早期团队只需要保留少数真正会影响行为的原则。

建议先从四条开始：

| 价值观 | 最小解释 |
|--------|----------|
| 求真 | 不包装问题，不隐瞒风险 |
| 客户价值 | 不做没有真实问题支撑的事情 |
| 闭环 | 问题必须有结论、有责任、有后续 |
| 长期质量 | 不用不可控技术债换短期漂亮结果 |

这四条足够覆盖大多数研发绩效场景。

每次复盘时，只问几个问题：

1. 我们有没有说真话；
2. 我们有没有解决真实问题；
3. 我们有没有把问题闭环；
4. 我们有没有透支长期能力。

如果这些问题能被持续认真回答，价值观就不是口号，而是团队的工作方式。

## 最终收束

研发绩效体系最后不能只落在 KPI 上。

KPI 能帮助我们观察状态，方法论能帮助我们管理过程，组织机制能帮助我们推动收敛，但价值观决定了团队在没有标准答案时如何选择。

研发团队真正长期有效的绩效体系，应该同时具备三件事：

1. 有目标，知道要解决什么问题；
2. 有方法，知道如何验证、收敛和交付；
3. 有价值观，知道什么行为值得坚持，什么行为不能被奖励。

所以，研发绩效的最后一层，不是指标，而是价值观。

它决定一支团队会不会为了短期数字牺牲真实问题，会不会为了个人表现隐藏风险，会不会为了完成任务透支长期能力，也决定这支团队能不能持续解决复杂问题。
