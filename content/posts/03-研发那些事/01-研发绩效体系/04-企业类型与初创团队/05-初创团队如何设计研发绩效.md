前面几篇讨论的是 KPI、研发绩效和方法论。但如果真正落到企业实践，会遇到一个更现实的问题：

**不同价值创造方式，绩效管理逻辑是不一样的。**

确定性运营、探索性创新、以及两者并存的双元混合，对绩效的要求并不相同。如果不区分价值创造方式，直接套用某一家大公司的绩效制度，很容易把团队带偏。

尤其是初创团队，更不能简单照搬大厂。大厂的绩效体系往往是在组织规模、业务成熟度、管理层级和人才密度都已经发展到一定阶段后形成的。初创团队如果过早引入复杂考核、强制分布、重流程和重汇报，反而会削弱最重要的东西：方向判断、快速验证、真实反馈和团队战斗力。

## 先区分三种价值创造方式

从绩效管理角度，与其说企业可以被绝对分成几类，不如说不同业务、不同阶段有不同的主导价值创造方式。

这三种方式更接近 MECE：分类对象不是“企业名称”，而是“当前业务状态”。同一家公司内部可以同时存在多种状态，但某个具体业务、项目或阶段，通常会有一种主导逻辑。

| 价值创造方式 | 核心问题 | 绩效管理重点 | KPI 的位置 |
|----------|----------|--------------|------------|
| 确定性运营 | 如何稳定、高效、低成本地交付 | 标准化、效率、成本、质量、规模复制 | 可以比较强 |
| 探索性创新 | 如何找到新问题、新方案、新产品 | 探索、验证、突破、工程实现 | 不能太早太强 |
| 双元混合 | 如何同时管理成熟业务和新业务 | 分类管理、组合评估、动态调整 | 分业务、分阶段使用 |

确定性运营更接近确定性执行，绩效指标可以更明确。比如产能、成本、质量、交付周期、客户满意度。

探索性创新面对的是不确定性问题。它的核心不是“把已知动作做快”，而是“找到正确问题和可行路径”。如果这个阶段 KPI 太强，团队会回避不确定性，选择容易完成、容易量化、容易汇报的事情。

双元混合最复杂。它既有成熟业务需要效率，也有新业务需要探索。如果用同一套绩效逻辑管理所有业务，就会出现两种错误：用确定性运营指标压死创新，或者用创新口号放松成熟业务纪律。

## 大疆、华为、腾讯分别代表什么

如果从中国优秀企业中选几个参照，大疆、华为、腾讯很有代表性。但它们不能被简单归为同一种绩效管理模型。

### 华为：大规模工程组织的绩效逻辑

华为更接近“大规模工程组织，以确定性运营为主，同时保留长期技术创新”。

它的核心不是单点创意，而是把客户价值、技术能力、组织执行、流程体系、全球交付和长期投入结合起来。这样的组织需要很强的目标承接、责任贡献评价、干部管理、激励约束和组织纪律。

华为的绩效逻辑对初创团队有几个启发：

1. 绩效必须对准客户价值，不能只对准内部任务；
2. 贡献要和责任匹配，不能只看忙不忙；
3. 组织必须能识别真正承担问题的人；
4. 短期产出不能透支长期能力。

但初创团队不能直接照搬华为。

华为的强绩效、强流程、强组织纪律，是建立在大规模组织和成熟管理体系上的。初创团队如果过早引入类似机制，容易变成“还没有规模，先有官僚；还没有稳定业务，先有复杂考核”。

对初创团队来说，华为值得学的是客户价值、责任意识和长期主义，不是复杂流程和强制排名。

### 腾讯：平台型互联网组织的绩效逻辑

腾讯更接近“平台型互联网组织，典型特征是双元混合”。

它既有成熟业务，也有创新业务；既有产品迭代，也有平台生态；既要用户价值，也要商业结果。这样的组织不能只靠单一 KPI，也不能完全靠自由探索。它需要目标管理、产品数据、用户反馈、协同机制、人才发展和长期投入共同发挥作用。

腾讯的绩效逻辑对初创团队有几个启发：

1. 用户价值应该进入绩效判断；
2. 产品和技术团队不能只看交付，也要看体验和反馈；
3. 目标需要动态复盘，不能年初定完就机械执行；
4. 人才发展和组织学习也是绩效体系的一部分。

但初创团队也不能直接照搬腾讯。

腾讯的职级体系、人才发展体系、业务协同机制和组织评价方式，适合大规模平台型公司。初创团队如果照搬，会导致大量管理动作超过团队承载能力。

对初创团队来说，腾讯值得学的是用户导向、产品反馈和人才成长，不是复杂职级和多层评价流程。

### 大疆：探索性创新到体系化运营的绩效逻辑

大疆更接近“早期以探索性创新为主，成长后叠加确定性运营能力”的硬科技公司。它的发展过程也说明：探索性创新做大后，必须从天才驱动、产品驱动，逐步走向组织驱动和体系驱动。

大疆早期最重要的是产品判断、工程能力、技术突破和极致体验。这类业务如果一开始就用确定性运营 KPI 管理，很可能做不出真正有突破的产品。因为硬科技创新往往需要长期积累、跨学科协同、反复试错和工程验证。

大疆的绩效逻辑对初创团队有几个启发：

1. 好产品不是靠指标堆出来的，而是靠真实问题、工程深度和极致追求做出来的；
2. 创新需要高人才密度和高质量判断；
3. 探索阶段不能被短期 KPI 绑死；
4. 但组织变大后，必须补上规则、流程、监督和管理能力。

大疆最值得初创团队学习的，不只是“创新”，还有它后来的反思：再强的产品团队，如果长期缺少基本规则、组织边界和管理体系，也会走向失序。

对初创团队来说，大疆值得学的是产品精神、工程能力和求真文化，同时也要警惕长期依赖少数天才和创始人直觉。

## 三家公司给初创团队的共同启发

把大疆、华为、腾讯放在一起看，会发现它们虽然绩效机制不同，但有几个共同点。

### 1. 都不是为了 KPI 而 KPI

华为强调客户价值和责任贡献，腾讯强调用户价值和产品反馈，大疆强调产品极致和工程真实。

它们的绩效体系背后，都有一个更底层的价值判断：企业到底靠什么创造价值。

所以初创团队设计绩效，不能先问“用什么 KPI”，而要先问：

```text
我们这家公司现在靠什么创造价值？
```

如果答案还不清楚，绩效体系就不能设计得太重。

### 2. 都把绩效和组织能力绑定

绩效不是单纯评价个人，而是在塑造组织能力。

华为通过绩效强化客户导向和责任贡献；腾讯通过目标、反馈和人才体系强化产品与平台能力；大疆通过产品、工程和管理体系强化硬科技能力。

初创团队也一样。绩效体系不是为了年底打分，而是为了让团队形成正确能力。

### 3. 都不能脱离企业阶段

同一家公司在不同阶段，绩效逻辑也会变化。

早期更需要探索和突破，增长期更需要组织协同，成熟期更需要效率和纪律。初创团队最大的问题，往往不是没有绩效，而是过早使用成熟企业的绩效方式。

## 初创团队应该怎么设计绩效

初创团队的绩效体系应该非常克制。

它不应该追求完整，不应该追求复杂，也不应该追求“看起来专业”。它首先要服务四件事：

1. 活下来；
2. 找到方向；
3. 快速验证；
4. 建立最小组织能力。

可以用一个简单框架：

```text
少量目标 + 阶段判断 + 验证机制 + 质量底线 + 行为反馈
```

## 第一层：公司层只保留少量北极星目标

初创团队不能有太多目标。

目标越多，资源越分散；资源越分散，越难形成突破。公司层应该只保留 1 到 3 个最关键目标。

不同阶段的北极星目标可以不同：

| 阶段 | 典型目标 |
|------|----------|
| 0->1 探索 | 找到真实需求、验证核心假设、完成关键原型 |
| 1->10 验证 | 形成可交付产品、拿到关键客户、验证商业闭环 |
| 1->100 规模化 | 提升交付效率、降低成本、稳定质量、扩大复制 |

公司层目标要少，但必须真实。不能把“做很多功能”“开很多会议”“完成很多任务”当成核心目标。

## 第二层：项目层按阶段管理

初创团队最常见的错误，是所有项目都用同一种管理方式。

探索项目、验证项目、规模化项目，应该有不同的绩效逻辑。

| 项目类型 | 应该评价什么 | 不应该只看什么 |
|----------|--------------|----------------|
| 探索项目 | 假设是否清楚，验证是否有效，学习是否发生 | 是否按原计划完成 |
| 验证项目 | 里程碑是否收敛，质量是否可控，客户是否认可 | Demo 是否好看 |
| 规模化项目 | 效率、稳定性、成本、复用能力 | 单次英雄式交付 |

这和前面讨论的研发绩效逻辑是一致的：越不确定，越要看验证；越确定，越可以看效率。

## 第三层：个人层不做强制低绩效比例

初创团队尤其不适合强制低绩效比例。

因为初创团队人数少、协作强、风险高。如果强行要求一定有人低绩效，团队会很快进入自我保护状态：少接难题、少做探索、多做容易证明自己的事情。

更合理的原则是：

**低绩效不是配额，而是诊断结果。**

初创团队应该识别低质量行为，而不是制造低绩效名额。

真正需要被指出的问题包括：

1. 长期没有有效产出；
2. 不做验证，只凭感觉推进；
3. 遇到风险不暴露；
4. 问题没有闭环；
5. 反复制造质量问题；
6. 只完成表面任务，不承担真实问题；
7. 透支团队长期能力换取短期结果。

如果团队整体都在高质量解决问题，就不需要为了“绩效分布”强行打低分。

## 第四层：用行为质量替代复杂评分

初创团队可以不用很复杂的评分模型，但必须有清晰的行为标准。

可以用下面的判断表：

| 行为 | 评价 |
|------|------|
| 主动识别关键问题，并推动验证闭环 | 高质量贡献 |
| 能在不确定性中快速学习和调整 | 高质量贡献 |
| 能把复杂问题工程化落地 | 高质量贡献 |
| 能暴露风险，并推动团队一起解决 | 高质量贡献 |
| 只完成被分配任务，不关心问题是否解决 | 普通贡献 |
| 没有验证、没有闭环、风险不透明 | 低质量行为 |
| 为短期结果隐藏质量问题或制造长期负债 | 低质量行为 |

这样评价的好处是，它不会把绩效简化成任务数量，而是把团队注意力拉回真正重要的行为。

## 第五层：绩效周期要短，反馈要及时

初创团队不适合一年才做一次绩效总结。

一年太长，等到年底再发现方向错了、协作坏了、质量崩了，已经来不及。初创团队更适合短周期反馈。

建议节奏是：

| 周期 | 内容 |
|------|------|
| 每周 | 目标进展、风险暴露、阻塞处理 |
| 每 2 到 4 周 | 假设验证、阶段结论、优先级调整 |
| 每季度 | 目标复盘、贡献评价、能力短板、角色调整 |
| 每半年 | 薪酬、股权、晋升、去留等重大决策 |

绩效反馈应该尽量靠近事实发生的时间。问题越早暴露，修正成本越低。

## 第六层：激励要和阶段匹配

初创团队的激励也不能只有奖金。

不同阶段要激励不同东西：

| 阶段 | 激励重点 |
|------|----------|
| 0->1 探索 | 高质量验证、关键突破、主动承担不确定性 |
| 1->10 验证 | 客户成功、产品可用、质量闭环、跨职能协同 |
| 1->100 规模化 | 效率提升、稳定交付、成本下降、能力复用 |

对于初创团队，长期激励比短期奖金更重要。真正关键的人，应该通过股权、期权、角色成长、关键项目机会来绑定，而不是只靠季度奖金。

但长期激励也不能泛化。它应该奖励真正承担关键问题、创造长期价值、提升组织能力的人。

## 初创团队最应该避免的错误

### 1. 过早大厂化

团队还很小，就开始搞复杂职级、复杂评分、复杂汇报、复杂审批。

这会让组织看起来规范，但行动变慢，真实问题被流程覆盖。

### 2. 过度 KPI 化

所有事情都要数字化，所有人都要背指标。

这会让团队倾向于做可量化但不一定重要的事，回避真正难但关键的问题。

### 3. 创新无纪律

反过来，初创团队也不能用“创新”掩盖混乱。

探索可以失败，但必须有假设、有验证、有结论；创新可以试错，但不能无限试错；方向可以调整，但不能没有复盘。

### 4. 创始人主观评价一切

早期团队很容易依赖创始人判断，这是正常的。

但如果所有评价都来自创始人的即时感受，团队会变得不可预期。初创团队至少要建立最小的事实基础：目标、证据、复盘、风险记录、贡献记录。

## 一套适合初创团队的简化模板

初创团队可以用下面这张表做季度绩效复盘。

| 模块 | 要回答的问题 |
|------|--------------|
| 公司目标 | 这个季度最重要的 1 到 3 个目标是什么 |
| 阶段判断 | 当前是探索、验证还是规模化 |
| 关键假设 | 本季度验证了什么 |
| 主要结果 | 哪些结果真正推动了公司进展 |
| 风险暴露 | 哪些风险被发现，是否闭环 |
| 质量情况 | 是否留下明显技术债或质量隐患 |
| 个人贡献 | 谁承担了关键问题，谁推动了闭环 |
| 能力沉淀 | 是否形成可复用能力 |
| 下一步 | 继续、停止、调整还是扩大投入 |

这张表不复杂，但足够让团队从“忙不忙”转向“有没有解决真实问题”。

## 最终建议

初创团队设计研发绩效，应该记住三句话：

1. 向华为学客户价值和责任贡献，但不要照搬强流程和强排名；
2. 向腾讯学用户反馈和人才成长，但不要照搬复杂职级和多层评价；
3. 向大疆学产品精神和工程求真，但不要长期依赖天才驱动和创始人直觉。

初创团队最适合的绩效体系，不是大厂绩效体系的缩小版，而是一套围绕生存、验证、交付和组织能力成长的轻量机制。

它应该足够简单，简单到团队愿意使用；也要足够锋利，锋利到能识别真实贡献和低质量行为。

最终，初创团队的绩效不是为了给人排序，而是为了帮助团队更快找到方向、更真实地面对问题、更持续地形成解决复杂问题的能力。
