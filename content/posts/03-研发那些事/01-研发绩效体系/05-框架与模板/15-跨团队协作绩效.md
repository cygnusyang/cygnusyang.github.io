现代研发很少是单一团队能完成的，往往需要多个团队协作——前端、后端、算法、测试、产品、设计、供应链……跨团队协作已经成为常态。

但前面的体系主要讲单一团队/项目的绩效管理，没有专门讨论跨团队协作的绩效如何设计。这一章我们补上这块内容。

## 跨团队协作的常见问题

跨团队协作中，最常见的问题有：

| 问题 | 表现 |
|------|------|
| 团队墙 | 各团队只关注自己的指标，不关心整体目标 |
| 甩锅 | 遇到边界问题，互相推诿 |
| 对齐成本高 | 开会很多，但真正对齐的很少 |
| 协同质量差 | 你等我，我等你，交付周期被拉长 |
| 功劳争夺 | 成功了抢功，失败了甩锅 |
| 没人对最终结果负责 | 都说"我这部分完成了"，但整体没成 |

这些问题的根源往往不是"人不好"，而是"绩效设计有问题"——如果每个团队都只看自己的指标，自然不会关心整体目标。

## 跨团队协作绩效的设计原则

### 原则 1：先有共同目标，再有团队指标

**不能**：先给每个团队定指标，再让大家"协作"。
**要**：先定共同目标，再看每个团队如何为这个目标做贡献。

**例子**：
- 不好的做法："后端 KPI 是接口交付速度，前端 KPI 是页面上线速度，测试 KPI 是缺陷数量"
- 好的做法："我们的共同目标是'3 个月内产品上线，且核心功能稳定可用'，每个团队的指标都要服务于这个目标"

### 原则 2：区分"主责团队"和"协作团队"

不是所有团队的责任都一样，要区分：

| 角色 | 责任 | 评价重点 |
|------|------|---------|
| 主责团队 | 对最终结果负责 | 整体目标达成、里程碑、协同质量 |
| 协作团队 | 对自己负责的模块负责 | 交付质量、响应速度、协同质量 |

**关键**：必须有一个团队是主责，不能"大家一起负责"——大家一起负责，就是没人负责。

### 原则 3：共同评价，不拆分结果

不要把项目结果简单拆分给各个团队（如"功劳各占 30%"），这样容易导致：
- 团队只关注自己的一亩三分地
- 遇到边界问题互相甩锅
- 为了自己的指标牺牲整体利益

**更好的做法**：
1. 先评价项目整体绩效（目标达成、过程质量）
2. 再看每个团队在项目中的贡献（不是拆分结果，而是看行为质量）
3. 评价团队时问：
   - 这个团队是否真的推动了项目前进？
   - 是否主动承担了边界问题？
   - 协同质量如何？
   - 是否沉淀了可复用的东西？

### 原则 4：收集多方面反馈

评价协作质量，不能只听一方说，要收集多方面的反馈：

- 主责团队对协作团队的反馈
- 协作团队对主责团队的反馈
- 协作团队之间的互相反馈
- 上下游团队的反馈
- 用户或客户的反馈

这样评价才更全面、更公平。

## 跨团队协作的绩效结构

跨团队协作的绩效评价，可以用这个四维结构：

| 维度 | 说明 | 评价重点 |
|------|------|---------|
| 共同目标对齐 | 是否真的理解和对齐了共同目标 | 目标理解、优先级对齐、决策质量 |
| 协同质量 | 协作是否顺畅，有没有内耗 | 响应速度、沟通效率、问题解决、信息同步 |
| 交付质量 | 自己负责的部分是否按时按质交付 | 按期交付、质量达标、依赖管理 |
| 整体贡献 | 对最终结果的贡献是什么 | 是否真的推动了整体目标、是否主动承担边界问题 |

### 1. 共同目标对齐

**为什么重要**：如果大家对目标理解不一致，做得越多，错得越多。

**评价要点**：
- 是否真的理解共同目标？
- 优先级是否对齐？
- 决策是否以共同目标为依据？
- 遇到冲突时，是否回到共同目标讨论？

**证据**：
- 目标对齐会议记录
- 决策过程（是否基于共同目标）
- 优先级调整的理由
- 冲突解决的方式

### 2. 协同质量

**为什么重要**：协同质量不好，会导致内耗、延期、质量问题。

**评价要点**：
- 响应速度：对方提的需求/问题，多久能响应？
- 沟通效率：沟通是否清晰，有没有来回扯皮？
- 问题解决：遇到问题是否主动推动解决，还是甩锅？
- 信息同步：关键信息是否及时同步，有没有信息差？
- 协作意愿：是否愿意帮助其他团队，还是只关注自己？

**证据**：
- 协作团队的反馈（很重要）
- 问题闭环速度
- 沟通记录（看是解决问题，还是扯皮）
- 有没有因为信息差导致的问题

### 3. 交付质量

**为什么重要**：每个团队的交付质量，会影响整体进度和质量。

**评价要点**：
- 按期交付：是否按约定时间交付？
- 质量达标：交付的东西是否符合质量标准？
- 依赖管理：是否考虑了对其他团队的影响？
- 变更管理：有变更时是否及时通知相关团队？

**证据**：
- 交付记录
- 缺陷和问题
- 变更通知记录
- 下游团队的反馈

### 4. 整体贡献

**为什么重要**：有些团队虽然自己的部分完成了，但对整体目标没什么贡献；有些团队可能做得不多，但关键时候推动了整体前进。

**评价要点**：
- 是否真的推动了整体目标？
- 是否主动承担了边界问题？
- 是否帮助其他团队解决了问题？
- 是否沉淀了可复用的东西？

**证据**：
- 关键决策中的贡献
- 边界问题的解决
- 其他团队的认可
- 沉淀的资产

## 跨团队协作的 KPI 设计

跨团队协作的 KPI，不要太复杂，要关注真正重要的东西。

### 推荐 KPI

| KPI | 计算方式 | 说明 |
|-----|---------|------|
| 共同目标达成率 | 共同目标中，达成的数量 / 总数量 | 这是最重要的 |
| 协作满意度 | 其他团队对这个团队的满意度评分 | 收集反馈 |
| 跨团队问题闭环速度 | 跨团队问题从提出到解决的平均时间 | 看协同效率 |
| 按期交付率 | 按约定交付的次数 / 总次数 | 看交付可靠性 |
| 边界问题主动承担率 | 团队主动承担的边界问题数 / 总边界问题数 | 看是否愿意担当 |

### 如何使用这些 KPI

**不要**：
- 用这些 KPI 机械打分
- 只看自己团队的指标，不看整体
- 因为某个 KPI 不达标就否定整体贡献

**要**：
- 用 KPI 发现问题（如"协作满意度低，为什么？"）
- 结合行为质量评价（不只是看数字）
- 关注趋势（是在变好还是变坏）
- 收集多方面反馈（不只是看数字）

## 跨团队协作的奖惩机制

### 奖励什么？

| 行为 | 说明 |
|------|------|
| 主动推动整体目标 | 不只是关注自己的指标，而是关心整体 |
| 主动承担边界问题 | 遇到没人管的问题，主动站出来 |
| 帮助其他团队 | 主动提供帮助，分享经验 |
| 高质量协同 | 响应快、沟通顺、不甩锅 |
| 沉淀跨团队协作机制 | 让以后的协作更顺畅 |

### 惩罚什么？

| 行为 | 说明 |
|------|------|
| 只关注自己指标，牺牲整体 | 为了自己的 KPI，让整体受损 |
| 甩锅 | 遇到问题推给别人 |
| 响应慢、不配合 | 其他团队找过来，半天没反应 |
| 隐瞒信息 | 关键信息不同步，导致其他团队踩坑 |
| 不接受反馈 | 其他团队提了意见，不改进 |

### 奖励机制建议

- **团队奖**：如果共同目标达成，所有协作团队都应该获得奖励（不是只有主责团队）
- **协作奖**：专门奖励那些协同质量好、主动帮助别人的团队
- **沉淀奖**：奖励那些沉淀了跨团队协作机制的团队
- **长期激励**：跨团队协作的价值往往长期才能体现，要配合长期激励

## 跨团队协作的治理机制

光有绩效评价还不够，还要有治理机制，让协作更顺畅。

### 1. 明确角色和责任

- 谁是主责团队？
- 每个团队的责任边界是什么？
- 关键决策谁来拍板？
- 遇到冲突谁来协调？

**不要**：责任模糊，都说"不归我管"。

### 2. 建立协作契约

协作开始前，各团队一起约定：
- 共同目标是什么？
- 每个团队的交付物是什么？
- 时间节点是什么？
- 质量标准是什么？
- 沟通机制是什么？（如每周同步会、响应时间要求）
- 变更流程是什么？
- 风险如何同步？

**写成文档**，大家达成共识，不要只是口头约定。

### 3. 定期同步

- **每周**：各团队同步进展、风险、问题
- **每月**：对齐目标、调整优先级
- **里程碑**：做阶段回顾和规划

**同步会的重点**：不是"汇报工作"，而是"暴露问题、解决问题、对齐目标"。

### 4. 冲突解决机制

跨团队协作难免有冲突，要有明确的冲突解决机制：

- **第一步**：双方先自己沟通，尝试解决
- **第二步**：如果解决不了，上升到双方负责人
- **第三步**：如果还解决不了，上升到共同上级
- **第四步**：最后由共同上级拍板

**关键**：不要让冲突卡太久，要有 escalation 路径。

### 5. 复盘和改进

每个跨团队项目结束后，都要做协作复盘：
- 哪些地方协作顺畅？为什么？
- 哪些地方协作不好？为什么？
- 我们学到了什么？
- 如何改进以后的协作？
- 哪些机制可以沉淀下来？

**不要**：项目结束就完了，下次还是同样的问题。

## 总结

跨团队协作绩效的核心原则是：

1. **先有共同目标，再有团队指标**：不要让团队只关注自己的 KPI
2. **明确主责和协作**：必须有一个团队对最终结果负责
3. **共同评价，不拆分结果**：先看整体，再看每个团队的贡献
4. **收集多方面反馈**：不只是听一方说
5. **配合治理机制**：角色、契约、同步、冲突解决、复盘

当跨团队协作的绩效设计真正鼓励"为整体目标做贡献"，而不是"只关注自己的指标"时，团队墙才会消失，协作才会真正顺畅。
